3. B2B Registration — How it works
It's nothing like normal shop sign-up
How did you handle user registration in the B2B project?
In a normal shop (B2C), anyone can sign up with their email. B2B is completely different.

The whole site is locked. If you're not logged in, every page redirects you to the login page — even the homepage. You literally cannot see anything without an account.

Accounts are not self-service. A company is created by the admin. Then the company admin adds users under that company. Regular users can't add more users or change addresses — only the company admin can.
Say this

"Registration is not open to the public. The entire site is locked behind login — if you visit any page without being logged in, you get redirected to the login page. We built this with a custom plugin that intercepts every request and checks if the user is authenticated."

"Account creation uses Magento's native B2B Company Accounts feature. The admin creates a company, and then the company admin manages who gets access. We added custom restrictions so that regular users can't create other users or edit company addresses — only the company admin can do that."

  • Module name: Scandiweb_LoginRestriction — a plugin on FrontControllerInterface
  • It checks $customerSession->isLoggedIn() on every request
  • If not logged in → redirect to /customer/account/login
  • Magento B2B Company Accounts = native Magento feature, not custom-built
  • Our custom part = restricting what regular users can do (RestrictUserCreation, RestrictAddressEdit plugins)

Don't say "users register themselves." They don't. The admin creates company accounts and the company admin manages users.

4. Preorder vs Stock Orders
Two completely separate Magento websites
What is the difference between preorder and stock in that project?
This is NOT about "out of stock" items. That's a common mistake — don't say preorder means ordering something that is out of stock. That's "backorder" in Magento. This is a completely different business concept.

How the sports industry works: brands design next season's products (say, next winter's ski boots) months before they exist. Retailers — the sports shops — need to commit early so the brand knows how many units to manufacture. So in spring, a retailer visits the preorder website and places a bulk order for next winter's collection. The products aren't in any warehouse yet. They'll be shipped when the season starts.

Then separately, the same retailer can visit the stock website and order products that are in the warehouse right now — available to ship immediately.

These are two completely separate Magento websites with the codes stock and preorder. Each has its own catalog, its own stores, and its own URL. The project also has a Sgd_Season module that links products to specific seasons — which is why the order history table shows a "Season" column so retailers know which season each preorder belongs to. After login, the store selector popup lets the buyer pick which website to shop on.
Stock OrdersPre-Orders

Say this

"The platform has two separate Magento websites — stock and preorder. Stock orders are for products available right now in the warehouse. Preorders are for the next season — retailers order months in advance before the products are even manufactured. It's how the sports industry works — you commit to your winter collection in spring."

"The order history page splits orders into 'Pre-Order Confirmations' and 'Stock Order Confirmations'. We detect which type an order is by checking which website it was placed on — the store's website code."

  • Preorder ≠ out of stock. Preorder = retailers ordering next season's products months before they're manufactured.
  • The business reason: the brand needs to know how many units to produce — retailer commitments tell them that.
  • Two Magento websites: stock (available now) and preorder (next season)
  • After login → store selector popup → pick which website to shop on
  • Sgd_Season module links products to seasons — shows in the "Season" column in order history
  • OrderConfirmations ViewModel checks $store->getWebsite()->getCode() to split orders into preorders vs stock orders
  • Magento's "backorder" = completely different thing = allowing orders when current stock qty is 0

6. What is Hyvä Theme?
The modern way to build Magento frontends
What is Hyvä and why use it?
The old Magento theme (Luma) uses old JavaScript tools — RequireJS and KnockoutJS — that were slow and hard to work with. Pages were heavy.

Hyvä replaces all of that with:
Alpine.js — simple JavaScript for interactions (like dropdowns, modals, cart updates)
TailwindCSS — utility CSS classes so you style directly in HTML instead of writing separate CSS files

The result: pages that score 90+ on Google Lighthouse (performance test) instead of 30.
Say this

"Hyvä is a modern Magento theme that replaces the old Luma frontend. Instead of RequireJS and KnockoutJS, it uses Alpine.js for JavaScript and TailwindCSS for styling. The result is much faster pages — on the projects I worked on, Lighthouse scores went from around 30 on Luma to 90+ on Hyvä, and the JavaScript bundle went from 800KB down to about 40KB."

  • Alpine.js = small, simple JS framework. Think of it like Vue but tiny.
  • TailwindCSS = write styles as HTML classes like class="flex gap-4 text-sm"
  • Lighthouse score = Google's performance rating (0-100). Higher is better.
  • 800KB → 40KB JS = 20× smaller. That's why pages load so much faster.
How do you set up a new Hyvä project from scratch?
The first sprint is all about design foundation — before any feature work.
Say this

"I start by setting up the design system. I configure tailwind.config.js with the brand's colors, typography scale, and breakpoints. Then I wire up a /style_guide page — it's a route in Magento that renders all the design tokens live: colors, headings, buttons, form fields. That way the whole team can verify the design before any feature is built."

7. How do you customize Magento without touching core files?
The most important Magento concept — you MUST know this
You never edit Magento's core files directly. If you do, your changes get wiped on the next update. Instead, you use these techniques:
1
Plugin (Interceptor) — run your code before, after, or around any Magento method. Example: before the login method runs, check if the user is allowed. This is what the LoginRestriction module uses.
2
Preference (Rewrite) — completely replace a Magento class with your own. Like saying "use MY version of this class instead." Use sparingly — can cause conflicts.
3
Observer/Event — Magento fires events at key moments (order placed, product saved). You listen for those events and run your code. No touching core needed.
4
Layout XML + Template override — copy a Magento template file into your theme folder with the same path. Magento uses yours instead. No editing the original.
Say this

"Magento has a layered extension system so you never touch core. The three main techniques are: plugins — to run code before or after any method; observers — to listen for Magento events like 'order placed'; and template overrides — copy a template into your theme at the same path and Magento uses yours automatically. I used plugins heavily on the B2B project, for example to intercept every page request and redirect guests to login."

10. Hardest Technical Challenge
The Algolia integration on Gemoss
Tell me about the hardest technical problem you solved.
The Algolia integration on Gemoss was complex because the module assumed a certain design (a search toggle button) but Gemoss had the search bar always visible. You had to rework the initialization logic to match.
Say this

"The Algolia search integration on Gemoss. The challenge wasn't just installing the module — it was making it work correctly inside Hyvä where the Gemoss design had the search bar always visible, with no toggle button. The Hyvä Algolia module assumed a toggle flow, so its initialization didn't fire correctly."

"I had to override the form template, safely merge the Algolia config without breaking values set elsewhere, and guard against the autocomplete being initialized twice. I also had to register the inline script with Hyvä's CSP system — Hyvä blocks inline scripts by default for security."

"Solving it taught me to always read the source of a third-party module before overriding its templates — the initialization order is never obvious from the outside."

  • CSP = Content Security Policy. Browser security that blocks inline scripts. In Hyvä you must call $hyvaCsp->registerInlineScript() to allow them.
  • window.algoliaConfig = a global JS object. Multiple templates write to it. You have to merge carefully so they don't overwrite each other.
  • Autocomplete guard = a boolean flag (autocompleteInitialized) so the init code only runs once even if triggered multiple times.

12. Infrastructure — Readymage, AWS, Cloudflare, GCP
Be honest about what you've used directly
What is GCP and have you used it?
GCP (Google Cloud Platform) is a service from Google where you rent servers, databases, and other infrastructure. You haven't directly provisioned GCP, but you understand the concepts through Readymage.

Readymage is Scandiweb's managed Magento hosting platform. It runs on AWS (Amazon's cloud — similar idea to GCP but from Amazon). Every Scandiweb project you worked on ran on Readymage — containerised PHP/Nginx, managed MySQL, Kubernetes auto-scaling.
Say this

"I haven't provisioned GCP directly, but I understand the concepts through Readymage — Scandiweb's managed Magento hosting platform built on AWS. All my Scandiweb projects ran on it: containerised PHP/Nginx, managed MySQL, Kubernetes auto-scaling. GCP covers the same concepts with different service names — Compute Engine instead of EC2, Cloud SQL instead of RDS. I'd get up to speed quickly."

What is Cloudflare and have you used it?
Cloudflare sits in front of your server. Every visitor goes through Cloudflare first. You used it on the J.R. Dunn Magento-to-Shopify migration.
Say this

"I used Cloudflare on the J.R. Dunn migration from Magento 1 to Shopify. It's a layer that sits in front of your server — it handles CDN (serving files fast from servers near the user), WAF (blocking attacks before they reach your server), DDoS protection, and free SSL certificates. For Magento the key thing is to exclude dynamic pages like checkout from Cloudflare's cache, otherwise users would see each other's cart data."

  • CDN = Content Delivery Network. Copies your images/CSS/JS to servers around the world. Users download from the nearest one — faster.
  • WAF = Web Application Firewall. Blocks hackers before they reach Magento.
  • DDoS = Distributed Denial of Service. Attackers send millions of fake requests to crash your server. Cloudflare absorbs them.
  • Always bypass Cloudflare cache for /checkout, /cart, /customer — these pages are personal to each user.
